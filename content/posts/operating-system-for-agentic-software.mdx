Most people use AI to code like this: they describe a feature in a sentence, let the agent generate a few hundred lines, skim the diff, and ship it. It works until it doesn't — and when it stops working, it stops badly. A vague prompt becomes a large uncontrolled change, which quietly breaks the architecture, which gets verified shallowly, which becomes impossible to review.

I build this way for real — Neural Edge, Neural OS, a booking platform, a few smaller products — and the difference between casually using AI and actually delivering with it isn't the model. It's the system around the model. So I wrote mine down.

## The failure mode nobody admits

The honest version of how AI-assisted development usually goes:

```text
vague prompt -> large uncontrolled change -> broken architecture -> shallow verification -> difficult review
```

Every step makes the next one worse. The agent guesses the product because you never defined it. It guesses the architecture because there were no contracts. It says "done" because nothing told it what done means. And by the time you're reviewing, the change is too big and too entangled to reason about.

The model isn't the problem. The absence of engineering discipline is. Coding agents are genuinely useful — but only when they're surrounded by the same controls a good team would demand from a human: clear intent, written context, contracts, tests, tool limits, small reviewable units, human review, and a retrospective.

## The pattern that actually works

The fix is to replace one big leap with a sequence of small, controlled steps:

```text
Intent -> Context -> Spec -> Contracts -> Tests/Evals -> Vertical Slice -> PR Loop -> Review -> Ship -> Retrospective
```

Each stage exists to remove a specific kind of guessing:

- **Intent** defines the problem, the user, what success looks like, and what's out of scope — before any code.
- **Context** forces the agent to read the existing system instead of inventing an architecture that conflicts with reality.
- **Spec** turns intent into testable behaviour: happy paths, failure paths, edge cases, acceptance criteria.
- **Contracts** give the agent hard interfaces — schemas, endpoints, tool signatures — to implement against, not vibes.
- **Tests and evals** define how correctness will be proven, written *before* the build, not bolted on after.
- **Vertical slice** ships the smallest end-to-end version that proves the core value with real architecture.
- **PR loop** expands the system in small, reviewable increments instead of one heroic commit.
- **Review, ship, retro** treat review as a control point and feed every lesson back into the process.

The whole point is to make the agent's work **bounded, inspectable, testable, and reversible**. If a change can't be reviewed or rolled back, the process failed regardless of whether the code runs.

## Five layers, not one system prompt

People try to control agents with a single clever system prompt. That's not enough — one prompt can't govern repository hygiene, tool permissions, and multi-agent handoffs at the same time. I think of it as five layers:

```text
Layer 1: Delivery Workflow      - turns an idea into scoped, testable, reviewable work
Layer 2: Agent Constitution     - how the agent plans, asks, verifies, respects boundaries
Layer 3: Execution & GitHub      - repos, branches, PRs, CI checks, merge authority
Layer 4: Skills & System Context - reusable project capabilities without bloating prompts
Layer 5: Multi-Agent Model       - orchestrators, workers, reviewers, handoffs, merge gates
```

The system is only *complete* when all five exist. But — and this matters — not every task needs all five running at full strength. A one-line copy fix doesn't need a threat model. A payments integration absolutely does.

## Right-size, or drown

The trap on the other side of "no process" is documentation theatre: writing an architecture decision record for a button colour. So the first decision on any task is how much ceremony it deserves.

```text
Light Mode     - small fixes, copy, simple refactors, low-risk UI
Standard Mode  - normal features, endpoints, UI flows, integrations
Full Mode      - high-risk, multi-agent, migrations, security, portfolio-grade work
```

Use the smallest mode that still gives you control. The goal isn't to produce documents — it's to make sure risky work never proceeds on a vague prompt, and trivial work never drowns in paperwork. (I go deeper on this in [Right-Size the Process](/blog/right-size-the-process).)

## Who owns what

The single rule that holds the whole thing together is a clear division of authority between the human, the software, and the agent:

```text
The human owns intent and risk.
The spec owns scope.
The contracts own interfaces.
The tests and evals own correctness.
The software owns deterministic control flow.
The agent handles ambiguity and implementation assistance.
The review loop decides whether the work is good enough.
```

Notice what the agent does *not* own: scope, correctness, or risk. Those stay with deterministic systems and with me. The agent is brilliant at the ambiguous middle — exploring a codebase, drafting an implementation, proposing tests, reviewing a diff. It should not be the thing that decides whether your migration is safe.

Two ideas in here are big enough to deserve their own treatment: that [the tools are the dangerous part, not the model](/blog/tools-are-the-dangerous-part), and that [you should write the tests before the agent writes the code](/blog/tests-before-the-agent-writes-code).

That's the difference between casually using AI to code and running a disciplined delivery system. The model keeps getting better. The discipline is what makes it safe to trust.

When people worry about AI agents, they worry about the model saying something wrong. That's the wrong fear. A wrong sentence is harmless. A wrong *tool call* is what deletes the row, emails the customer, or charges the card.

This is the part of [my agentic delivery system](/blog/operating-system-for-agentic-software) I'm most strict about, because it's where the real damage lives.

## Where the real risk lives

Text generation is reversible — you read it, you discard it. Tool use is where an agent reaches out and changes the world. So the most serious risk in any agentic system comes from tools, not from tokens. Anything that does one of these deserves suspicion:

```text
sends external communication
modifies production data
deletes files
deploys code
charges money
updates customer records
calls a third-party API
changes permissions
```

Each of those is a one-way door. The model proposing to walk through it is fine. The system letting it walk through unsupervised is not.

## Classify every tool

Before an agent can touch a tool, that tool gets a risk level. This single step removes most surprises:

```text
Low       read-only / local-only      read file, search repo, run tests     no approval
Medium    local mutation, reversible  edit code, create file, new branch     sometimes
High      external mutation / data    send email, write DB, deploy, API      yes
Critical  irreversible / financial    payment, deletion, prod migration      mandatory
```

The rule that follows is simple: approval requirements scale with reversibility. Reading a file needs nothing. Deploying needs a human. The agent doesn't get to decide which bucket a tool is in — I do, up front.

## The tool contract

Every non-trivial tool gets a contract, the same way an API does. Vague tools are how agents cause accidents.

```text
Purpose            what it does
Risk level         low / medium / high / critical
Input schema       validated, not free text
Output schema      structured, checkable
Permissions        what it may access
Forbidden uses     what it must never do
Approval           yes / no / conditional
Dry-run            available / not
Logging            what gets recorded (never secrets)
Failure behaviour  what happens on error
Idempotency        how retries and duplicates are handled
```

The two fields people skip — **dry-run** and **idempotency** — are the ones that save you. A dry-run lets a high-risk action be inspected before it fires. An idempotency key means a retried payment doesn't become two payments.

## Read before you write

A pattern I enforce everywhere: agents explore in read-only mode before they're allowed to mutate. Let the agent search the repo, read the schema, run the tests, and propose a plan — all safely — before any tool with write access is on the table. Most mistakes are caught in the gap between "here's what I'll do" and actually doing it.

## Treat model output as a proposal

The mental model that ties it together:

> Every LLM output is a proposal, not trusted executable truth.

Model text should never flow directly into a tool call without validation in between. You validate the input against a schema, check it against business rules, and — for anything irreversible — put a human in the loop. The agent suggests; deterministic code and I decide. That's not a lack of trust in the model. It's the same discipline you'd apply to any system component that can reach production.

Get this layer right and agents become genuinely safe to run. Get it wrong and it only takes one unsupervised tool call to find out.

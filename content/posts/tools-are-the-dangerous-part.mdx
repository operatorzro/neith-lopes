When people worry about AI agents, they picture the model saying something wrong. That is the wrong fear. A wrong sentence is harmless. A wrong tool call is what deletes the row, emails the customer, or charges the card.

This is the part of [my agentic delivery system](/blog/operating-system-for-agentic-software) I am strictest about, because it is where the real damage lives.

## Where the real risk lives

Text generation is reversible. You read it, you discard it. Tool use is where an agent reaches out and changes the world, so the serious risk lives there, not in the tokens. Anything that does one of these deserves suspicion:

```text
sends external communication
modifies production data
deletes files
deploys code
charges money
updates customer records
calls a third-party API
changes permissions
```

Each one is a one-way door. The model proposing to walk through it is fine. The system letting it walk through unsupervised is not.

## Classify every tool

Before an agent can touch a tool, that tool gets a risk level. This one step removes most surprises:

```text
Low       read-only / local-only      read file, search repo, run tests     no approval
Medium    local mutation, reversible  edit code, create file, new branch     sometimes
High      external mutation / data    send email, write DB, deploy, API      yes
Critical  irreversible / financial    payment, deletion, prod migration      mandatory
```

The rule that follows is simple: approval scales with reversibility. Reading a file needs nothing. Deploying needs a human. The agent does not decide which bucket a tool sits in. I do, up front.

## The tool contract

Every non-trivial tool gets a contract, the same way an API does. Vague tools are how agents cause accidents.

```text
Purpose            what it does
Risk level         low / medium / high / critical
Input schema       validated, not free text
Output schema      structured, checkable
Permissions        what it may access
Forbidden uses     what it must never do
Approval           yes / no / conditional
Dry-run            available / not
Logging            what gets recorded (never secrets)
Failure behaviour  what happens on error
Idempotency        how retries and duplicates are handled
```

The two fields people skip are the ones that save you: dry-run and idempotency. A dry-run lets a high-risk action be inspected before it fires. An idempotency key means a retried payment does not become two payments.

## Read before you write

A pattern I enforce everywhere: agents explore in read-only mode before they can mutate anything. Let the agent search the repo, read the schema, run the tests, and propose a plan, all of it safely, before any tool with write access is on the table. Most mistakes get caught in the gap between "here is what I will do" and actually doing it.

## Treat model output as a proposal

> Every model output is a proposal, not trusted executable truth.

Model text should never flow straight into a tool call. You validate it against a schema, check it against business rules, and for anything irreversible you put a human in the loop. The agent suggests. Deterministic code and I decide. That is not distrust of the model. It is the same discipline you would apply to any component that can reach production.

Get this layer right and agents become safe to run. Get it wrong and one unsupervised tool call is enough to find out.
